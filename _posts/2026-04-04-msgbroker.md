# Message Broker Comparison for Seismic Data Communication
## Earthworm 기반 지진 데이터 통신 시스템 확장을 위한 메시지 브로커 비교

> **배경**: [Earthworm](http://www.earthwormcentral.org/)은 지진 데이터 수집·처리를 위한 표준 시스템으로,  
> 내부적으로 공유 메모리 링 버퍼(Transport Ring)를 사용합니다.  
> 이를 범용 메시지 브로커로 교체하거나 연동하면 **성능·확장성·운영 편의성**을 크게 높일 수 있습니다.

---

## 📋 핵심 요구사항 (지진 데이터 특성 기준)

| 요구사항 | 설명 |
|---|---|
| **초저지연** | P파·S파 실시간 감지, 경보 발령에 수 ms 이내 전달 필요 |
| **고처리량** | 수백~수천 채널의 연속 파형 데이터(수 MB/s) 처리 |
| **시계열 순서 보장** | 파형 데이터의 시간 순서가 반드시 유지되어야 함 |
| **내결함성** | 네트워크 장애 시에도 데이터 유실 최소화 |
| **분산 확장** | 지역 거점 서버 ↔ 중앙 데이터센터 간 원격 통신 |

## 📊 항목별 상세 비교

### 1. 지연시간 (Latency)

> 지진 조기경보는 P파 감지 후 수 초 내에 경보 발령이 필요합니다.

| 브로커 | P50 지연 | P99 지연 | 비고 |
|---|---|---|---|
| **NATS** | < 0.1 ms | < 1 ms | 최상급 — 경보 발령에 최적 |
| **RabbitMQ** | ~1 ms | ~5 ms | 충분히 실용적 |
| **ActiveMQ** | ~5 ms | ~20 ms | 조기경보에는 다소 높음 |
| **Kafka** | ~5–15 ms | ~50 ms | 배치 최적화로 지연 발생, 튜닝 필요 |

---

### 2. 처리량 (Throughput)

> 광대역 지진계 1채널 = 100~500 sps × 4 bytes = 최대 2 KB/s.  
> 채널 수가 수백~수천 개일 경우 초당 수 MB ~ 수 GB 규모.

| 브로커 | 단일 노드 처리량 | 클러스터 확장 | 비고 |
|---|---|---|---|
| **NATS** | ~50M msg/s | 선형 확장 | 파형 샘플 단위 전송에 유리 |
| **RabbitMQ** | ~수백만 msg/s | 페더레이션으로 확장 | 충분하나 수천 채널에서 튜닝 필요 |
| **ActiveMQ** | ~수십만 msg/s | 제한적 | 대규모 채널 환경에서 병목 가능 |
| **Kafka** | ~수백만~수천만 msg/s | 파티션 추가로 선형 확장 | 대용량 스트리밍에 최강 |

---

### 3. 내결함성 & 데이터 영속성

| 브로커 | 영속성 방식 | 장애 복구 | 재처리 가능 여부 |
|---|---|---|---|
| **NATS** | 인메모리 기본, JetStream으로 파일 기반 영속성 추가 | 클러스터 자동 리더 선출 | ✅ JetStream 사용 시 |
| **RabbitMQ** | 디스크 큐, 미러링 큐 (Quorum Queue) | 미러링으로 복제 | ⚠️ Dead Letter Queue 활용 |
| **ActiveMQ** | KahaDB / JDBC 기반 영속성 | Master-Slave 구성 | ⚠️ 제한적 |
| **Kafka** | 분산 로그 파일 (Segment) | 복제 파티션(replication factor) | ✅ 기본 기능 — 파형 재분석에 최적 |

---

### 4. 운영 복잡도 & 생태계

| 브로커 | 설치 난이도 | 관리 도구 | 언어 SDK | 커뮤니티 |
|---|---|---|---|---|
| **NATS** | ⭐ 단일 바이너리, 의존성 없음 | NATS CLI, Surveyor | Go, Python, C, Java, 등 | 활발 |
| **RabbitMQ** | ⭐⭐ Erlang 런타임 필요 | 웹 관리 콘솔 내장 | 거의 모든 언어 | 매우 활발 |
| **ActiveMQ** | ⭐⭐ JVM 필요 | ActiveMQ 웹 콘솔 | JVM 중심, 일부 타 언어 | 성숙, 다소 정체 |
| **Kafka** | ⭐⭐⭐⭐ ZooKeeper 또는 KRaft 필요 | Kafka UI, Confluent Control Center | 거의 모든 언어 | 매우 활발 |

## 🎯 지진 데이터 시스템별 추천 시나리오

```
Earthworm Ring Buffer
        │
        ├──► NATS JetStream ──► 실시간 경보 발령 시스템 (초저지연)
        │                   └──► 현장 모니터링 대시보드
        │
        └──► Apache Kafka ──────► 파형 장기 저장 및 재분석
                            └──► ML 기반 이벤트 감지 파이프라인
```

| 사용 목적 | 추천 브로커 | 이유 |
|---|---|---|
| **실시간 지진 조기경보** | NATS | 최저 지연, 운영 단순, 경보 채널에 최적 |
| **연속 파형 스트리밍 저장** | Kafka | 대용량 로그 보존, 시간 기반 재처리 |
| **엔터프라이즈 데이터 라우팅** | RabbitMQ | 유연한 라우팅 규칙, 안정성 |
| **레거시 Java 시스템 연동** | ActiveMQ | JMS 표준 호환, 기존 시스템 재사용 |
| **NATS + 영속성 동시 필요** | NATS JetStream | 단일 시스템으로 두 요구사항 충족 |

## ⚠️ Earthworm 연동 시 고려사항

1. **Transport Ring → 브로커 Bridge 모듈 개발 필요**  
   Earthworm의 `export` / `import` 모듈 또는 커스텀 모듈로 메시지 변환

2. **메시지 포맷 표준화**  
   Earthworm 내부 메시지(TYPE_TRACEBUF2 등)를 MiniSEED, JSON, Protobuf 등으로 변환 권장

3. **시계열 타임스탬프 정밀도**  
   GPS 동기화 기반 µs 단위 타임스탬프를 메시지 헤더에 반드시 보존

4. **네트워크 파티션 대응**  
   원격 거점-중앙 간 WAN 지연을 고려한 QoS 설정 및 로컬 버퍼링 전략 수립
