## 1. 한눈에 보는 프로토콜 비교표

프로토콜을 크게 **통신·제어용**과 **방송·스트리밍용**으로 구분하면 선택이 명확해집니다.

| 프로토콜 | 주요 용도 | 지연 시간 | 전송 방식 | 장점 | 단점 | ⚠️ 주의사항 |
|:---:|:---:|:---:|:---:|---|---|---|
| **SIP** | 인터넷 전화(VoIP), 회의 시스템 | 매우 낮음 (< 150ms) | TCP / UDP | 세션 관리에 특화. 기존 전화망과 호환성 우수 | 미디어 전송 불가 (별도로 RTP 필요). 방화벽 설정이 복잡함 | NAT 환경에서 SIP ALG 설정 주의 |
| **RTP / RTCP** | 실시간 음성·영상 전송 | 매우 낮음 (< 150ms) | UDP | 순서 번호·타임스탬프로 동기화 가능. 지연에 최적화 | 자체 신호(Signaling) 기능 없음. 반드시 SIP 같은 제어 프로토콜과 함께 사용 | UDP 특성상 패킷 손실 발생 가능. RTCP로 품질 모니터링 필수 |
| **RTSP** | CCTV, IP 카메라 스트리밍 | 낮음 (< 1초) | TCP (제어) / UDP (데이터) | 재생·일시정지·탐색 등 VCR식 제어 가능. 보안 장비 표준 | HTTP 기반이 아니라 방화벽 통과 어려움. 브라우저 기본 지원 없음 | 일반 웹 서비스에는 부적합. 폐쇄망 환경에서 사용 권장 |
| **RTMP** | 방송 송출 (OBS → 서버) | 중간 (2 ~ 5초) | TCP | 낮은 송출 지연. 방송 소프트웨어(OBS 등)에서 광범위하게 지원 | Flash 기반 레거시 기술. 브라우저 네이티브 재생 불가 | Adobe Flash 공식 종료(2020). 시청용이 아닌 **송출 전용**으로만 사용 |
| **HLS** | 대규모 방송 시청 (서버 → 시청자) | 높음 (5 ~ 30초) | HTTP / HTTPS | CDN 친화적. 수백만 명 동시 시청 대응. 모든 브라우저·기기 호환 | 지연 시간이 길어 실시간 인터랙션 불가 | 짧은 Chunk 크기로 지연 줄일 수 있으나 서버 부하 증가 |
| **WebRTC** | 화상 채팅, 원격 제어, P2P 통신 | 극도로 낮음 (< 500ms) | UDP (P2P) | 플러그인 없이 브라우저만으로 동작. 서버 부하 최소화. 암호화 내장 | 다수 동시 접속 시 P2P 구조의 한계 (SFU 서버 필요). 방화벽 설정 복잡 | 대규모 방송에는 부적합. 소규모 양방향 통신에 최적 |

## 2. 고전 프로토콜 상세 설명 (RTP, RTCP, RTSP, SIP)

> 💡 이 프로토콜들은 주로 **폐쇄형 네트워크(기업망, CCTV 시스템, VoIP)**에서 여전히 핵심 기술로 사용됩니다.

### 🔹 RTP — 실시간 데이터 전송의 조상

```
[송신 측] → RTP 패킷 (순서번호 + 타임스탬프 + 데이터) → [수신 측]
```

- 음성·영상 데이터를 UDP 패킷으로 쪼개어 전송하는 **표준 규격**입니다.
- 패킷마다 **순서 번호(Sequence Number)** 와 **타임스탬프(Timestamp)** 가 붙어 있어, 도착 순서가 뒤바뀌거나 일부가 손실되어도 수신 측에서 올바르게 복원할 수 있습니다.
- 혼자서는 아무것도 못 합니다. 반드시 **SIP** 또는 **RTSP** 같은 시그널링 프로토콜과 함께 동작합니다.

### 🔹 RTCP — RTP의 품질 관리자

```
[수신 측] → RTCP 리포트 (패킷 손실률, 지터, RTT) → [송신 측] → 화질 조정
```

- RTP와 항상 쌍으로 동작하는 **품질 제어 프로토콜**입니다.
- 주기적으로 **패킷 손실률**, **지터(Jitter, 도착 시간 편차)**, **왕복 지연(RTT)** 등을 리포트합니다.
- 이 리포트를 바탕으로 송신 측이 비트레이트를 실시간으로 조절합니다.

### 🔹 RTSP — 스트리밍 서버의 리모컨

```
클라이언트 → RTSP PLAY   → 서버  (재생 시작)
클라이언트 → RTSP PAUSE  → 서버  (일시정지)
클라이언트 → RTSP TEARDOWN → 서버 (연결 종료)
```

- HTTP와 비슷하게 생긴 명령어로 동영상 서버를 **원격 제어**하는 프로토콜입니다.
- **CCTV/IP 카메라** 에서 RTSP URL(`rtsp://192.168.0.10/stream`)로 영상을 받아오는 방식이 표준입니다.
- 단, 실제 영상 데이터는 RTSP가 전송하지 않고 **RTP** 가 담당합니다. (RTSP는 제어만 합니다.)

### 🔹 SIP — 인터넷 전화의 교환원

```
발신자 → SIP INVITE  → 수신자  ("연결할까요?")
수신자 → SIP 200 OK  → 발신자  ("좋아요!")
         ↓ 이 시점부터 RTP로 실제 통화 시작
발신자 → SIP BYE     → 수신자  ("끊겠습니다.")
```

- 전화를 **연결하고 끊는 신호 처리(Signaling)** 만 담당합니다. 실제 음성·영상은 RTP가 처리합니다.
- 기업용 IP-PBX(사내 전화 시스템), 콜센터 솔루션의 핵심 기술입니다.
- SIP + RTP 조합은 **기존 공중 전화망(PSTN)** 과 연동이 가능하다는 강력한 장점이 있습니다.

## 3. 유튜브 스트리밍의 핵심 — RTMP vs HLS

유튜브 라이브 방송은 두 기술이 **파이프라인 형태로 연결**되어 동작합니다.

```
[유튜버 PC / OBS]
       │
       │  RTMP (송출, 낮은 지연)
       ▼
[유튜브 Ingest 서버]
       │
       │  영상 처리 + HLS 변환 (잘게 쪼개기)
       ▼
[CDN (전 세계 엣지 서버)]
       │
       │  HLS (시청, HTTP)
       ▼
[전 세계 수백만 시청자]
```

### 📤 RTMP — 송출(Ingest)의 표준

- OBS, Streamlabs 등 방송 소프트웨어가 **유튜브 서버로 영상을 밀어 넣을 때** 사용합니다.
- TCP 기반으로 **패킷 손실 없이 안정적으로** 서버에 도달합니다.
- Flash 시대에 만들어진 레거시 기술이지만, 방송 **송출 표준**으로는 현재도 압도적으로 사용됩니다.

> ⚠️ **주의**: RTMP는 시청자에게 직접 배포(Delivery)하는 용도로는 사용하지 않습니다.

### 📥 HLS — 시청(Delivery)의 표준

- 서버가 영상을 **수 초짜리 작은 조각(`.ts` 파일)**으로 잘라 HTTP로 제공합니다.
- 시청자 기기는 **재생 목록 파일(`.m3u8`)** 을 주기적으로 받아 다음 조각을 미리 내려받습니다.
- HTTP 기반이기 때문에 **CDN**과 완벽하게 호환되어 수백만 명이 동시에 시청 가능합니다.
- **ABR(Adaptive Bitrate)** 기능으로 인터넷 속도에 따라 화질을 자동으로 조절합니다.

> ⚠️ **주의**: 기본 HLS 지연 시간은 10~30초입니다. 지연을 줄이려면 **Low-Latency HLS(LL-HLS)** 사용을 고려하세요 (Apple 표준, ~2초 지연 가능).

## 4. WebRTC — 브라우저의 혁신

```
[브라우저 A] ←──── P2P (직접 연결) ────→ [브라우저 B]
                        ↑
              (STUN/TURN 서버로 경로 탐색 후
               실제 데이터는 직접 전송)
```

- 별도 프로그램 설치 없이 **브라우저만으로** 음성·영상·데이터를 실시간 전송합니다.
- **DTLS + SRTP** 암호화가 기본 내장되어 있어 별도 보안 설정이 필요 없습니다.
- 기본 구조는 **P2P** 이지만, 다자 회의(3명 이상)에서는 **SFU(Selective Forwarding Unit)** 서버를 통해 확장합니다.

### WebRTC 연결 과정 (간략)

1. **시그널링(Signaling)**: SDP(Session Description Protocol)로 서로의 미디어 능력을 교환 (WebSocket 등 별도 채널 사용)
2. **ICE(Interactive Connectivity Establishment)**: STUN/TURN 서버를 이용해 방화벽을 뚫고 최적 경로 탐색
3. **DTLS 핸드셰이크**: 암호화 키 교환
4. **SRTP 스트리밍**: 실제 미디어 전송 시작

> ⚠️ **주의**: 1:1 또는 소규모 통신에는 P2P로 충분하지만, **50명 이상 동시 접속**이 필요하다면 mediasoup, Janus 같은 SFU/MCU 서버 도입이 필요합니다.

## 5. 상황별 기술 선택 가이드

> 실무에서는 요구사항 한 줄만 명확하면 기술 선택이 거의 자동으로 결정됩니다.

### 🏗️ 어떤 시스템을 만드나요?

```
지연 시간이 얼마나 중요한가?
│
├─ 0.5초 미만 (실시간 반응 필수)
│   └─ ✅ WebRTC
│       예) 화상 회의, 원격 제어, 실시간 게임 스트리밍
│
├─ 2~5초 (약간의 지연 허용)
│   └─ ✅ RTMP (송출) + RTMP 재생 또는 WebRTC
│       예) 라이브 커머스, 인터랙티브 방송
│
├─ 10~30초 (대규모 동시 시청)
│   └─ ✅ RTMP (송출) + HLS (시청)
│       예) YouTube, Twitch, 대규모 스포츠 중계
│
└─ 폐쇄망 / 하드웨어 장비 연동
    └─ ✅ RTSP + RTP (CCTV, VoIP는 SIP + RTP)
        예) IP 카메라 관제, 기업 전화 시스템
```

### 📋 체크리스트

| 요구사항 | 권장 스택 |
|---|---|
| 브라우저에서 1:1 화상 통화 | WebRTC |
| 브라우저에서 10명 이하 그룹 화상 회의 | WebRTC + SFU (mediasoup 등) |
| 1만 명 이상 동시 라이브 스트리밍 | RTMP 송출 → HLS 배포 |
| IP 카메라(CCTV) 영상 수신 | RTSP + RTP |
| 기업용 인터넷 전화 구축 | SIP + RTP |
| 모바일 앱 실시간 방송 송출 | RTMP |
| 지연 최소화 라이브 (LL-HLS 적용) | RTMP 송출 → LL-HLS 배포 |

## 6. 자주 묻는 질문 (FAQ)

**Q. RTMP는 Flash가 죽었는데 아직도 써도 되나요?**

> A. **송출 용도로는 당분간 계속 사용됩니다.** Flash 종료는 브라우저에서 Flash 플러그인으로 RTMP 영상을 *재생*하는 방식이 막힌 것입니다. OBS → 서버로 *밀어 넣는 인제스트* 용도는 YouTube, Twitch 모두 여전히 RTMP를 표준으로 지원합니다. 단, 브라우저 재생에는 절대 사용하지 마세요.

**Q. HLS 지연이 너무 긴데 줄일 수 있나요?**

> A. **LL-HLS(Low-Latency HLS)** 로 약 2~3초까지 줄일 수 있습니다. 또는 지연에 민감한 서비스라면 **WebRTC 기반 CDN**(Cloudflare Stream, Mux 등)을 검토하세요.

**Q. WebRTC는 왜 방화벽에서 문제가 생기나요?**

> A. WebRTC는 UDP를 사용하는 P2P 통신이기 때문에 방화벽이 경로를 차단하는 경우가 많습니다. 이때 **STUN 서버**로 공인 IP를 확인하고, 그래도 안 되면 **TURN 서버**를 통해 중계합니다. 프로덕션 환경에서는 반드시 TURN 서버 구성이 필요합니다.

**Q. RTSP 영상을 웹 브라우저에서 보고 싶어요.**

> A. 브라우저는 RTSP를 기본 지원하지 않습니다. 중간에 **미디어 서버(예: MediaMTX, Wowza)**를 두고 RTSP → WebRTC 또는 RTSP → HLS로 변환하는 방식을 사용하세요.

```
┌──────────────────────────────────────────────────────────┐
│                    애플리케이션 레이어                      │
│  화상 회의    라이브 방송    CCTV 관제    VoIP 전화         │

       │           │              │           │
   WebRTC    RTMP → HLS       RTSP         SIP
       │                          │           │
       └──────────────────────────┴───────────┘
                                  │
                              RTP / RTCP
                         (실제 미디어 데이터 전송)
                                  │
                            UDP / TCP
```

> 📝 **정리**: 전통적인 SIP/RTP는 *통화*, RTMP/HLS는 *방송*, WebRTC는 브라우저 안에서 이 모든 것을 웹 기술로 통합한 가장 진화된 형태입니다. 어떤 기술을 쓸지 망설인다면 **"지연 시간"** 과 **"동시 접속자 수"** 두 가지 기준만 먼저 정하면 선택이 자연스럽게 따라옵니다.
